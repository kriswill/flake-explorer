The Rust port

flake-explorer's extractor, HTTP server, and static exporter in Rust — how it was built, what it trades, and what should come next.

How the port was done

Everything except the Svelte SPA was ported module-per-module into rust/, each file naming its TypeScript counterpart (rust/src/manifest.rs ↔ src/extract/manifest.ts). The schema came first: src/schema.ts became serde types with camelCase renames and omit-when-absent optionals, because the exercise stands or falls on byte-compatible JSON shape — the SPA consumes the same documents either way and cannot tell which implementation produced them.

The Nix side is shared, not ported. extract.nix is embedded in the binary and materialized into the user's cache directory keyed by content hash, so both implementations evaluate literally the same Nix code. The port's real surface is orchestration, parsing, and serving.

Highlighting went native. web-tree-sitter's WASM grammars were replaced by native tree-sitter linked with the same vendored highlight queries. One contract detail mattered: token offsets are UTF-16 code units (the client slices JS strings with them), so byte offsets are converted before runs are emitted. Output was verified byte-identical.

The Svelte build stays in bun. A bundle script emits an app-dist/ (JS, CSS, theme tokens, About data) that the binary locates at runtime — env var, exe-relative, repo checkout, or the Nix package's share/flake-explorer/app-dist. serve --dev shells back out to bun for rebuilds and pushes the same SSE reload.

Verification was diff-based. Both extractors ran against real flakes (including a 57-input NixOS configuration); manifests compared field-for-field, the full options blob loc-by-loc. The one true divergence became a fix: nixpkgs sometimes sets defaultText = false, which untyped JSON.parse passed silently — the Rust deserializer now coerces stray scalars deliberately instead of failing the chunk.

Tests were ported, not re-imagined. The real-nix mini-flake suite runs against the same fixture; the serve and degradation suites reuse the scripted-nix-shim technique. The crate split into lib + thin bin so tests/ can link, and shim suites hold one test per file — env/PATH mutation is process-global, and per-file test processes are what make it race-free.

Packaging is crane; coverage is cargo-llvm-cov. buildDepsOnly compiles the dependency tree as its own cached derivation, so CI rebuilds only the crate on source changes. Coverage runs twice: sandboxed as a flake check (build gate), and on the CI runner where real nix exists so the integration tests count — reported through octocov with its own artifact datastore.

Pros and cons

Pros

  • Single native binary — no bun runtime on the serving path; millisecond startup; no WASM init.
  • The type system audits the data — serde surfaced a real upstream quirk the untyped path hid.
  • Compiler-checked concurrency — single-flight extraction and manifest swaps use RwLock/watch/broadcast, not convention.
  • CI economics — crane's dep layer caches; source changes rebuild one crate, not 300 dependencies.
  • Verified parity — stronger starting evidence than most rewrites: same manifest, same blobs, same tokens.

Cons

  • Two implementations of one contract — schema.rs mirrors schema.ts by hand; drift is possible.
  • Two toolchains — the SPA still needs bun; contributors face Rust + TypeScript.
  • Extraction is no faster — wall time is bound by nix eval subprocesses either way.
  • Deliberate divergences — rs- fingerprint and a different safeName hash make data dirs per-implementation.
  • serve --dev is checkout-only — live reload shells out to bun and watches app/.

What full publishability still needs

  1. npm distribution — platform packages (-linux-x64, -linux-arm64, -darwin-arm64, -darwin-x64) under optionalDependencies, a small launcher shim as the bin entry that resolves the platform binary and points FLAKE_EXPLORER_APP_DIST at the main package's app-dist/. The published binaries must be portable builds (static musl for Linux, macOS-runner builds for Darwin) — the nix-built binary links glibc from /nix/store and runs nowhere else. Publish with --provenance; macOS needs only the linker's automatic ad-hoc signature; WASM is a non-starter (the program spawns nix/git and binds TCP — neither exists in WASI).
  2. Version single-sourcing — the crate version and About data must follow package.json before any release ships.
  3. Release workflow integration — a four-target build matrix, five npm publishes, and an explicit rollout decision: a dist-tag soak before switching the default, or a major-version cutover. Verified parity argues for a short soak.
  4. Docs and packaging parity — README/CLI docs and the flake's packages.default still point at the bun package; both are decisions to make explicitly.
  5. Windows, or an explicit “no” — neither implementation targets it; saying so costs one sentence.

Opinion: what to improve next

Performance

The honest ranking. First, reduce nix invocations, not Rust time: every options chunk is a full nix eval process. Batching sibling chunks into one eval returning per-chunk tryEval envelopes would cut process count severalfold without losing the degrade ladder — and the change lives in extract.nix, benefiting both implementations. Second, a persistent evaluator via the Nix C API would eliminate subprocess + JSON round-trips entirely, but couples the binary to a specific Nix and forfeits the “your nix, your store paths” property — an opt-in backend, never the default. Third, server niceties: pre-compress the page, ETag on manifest.json, concurrent path-info calls.

Maintainability

End the hand-mirrored schema: generate schema.ts from the Rust types (ts-rs) or validate both against one JSON Schema in CI. Promote the one-time verification into a permanent parity check — a CI job running both extractors against the mini-flake fixture and diffing normalized output turns “verified once” into “verified every commit,” and is the safety net for eventually retiring the bun extractor. And simplify the options worker pool into a channel-based work-stealing pattern now that the language offers better tools than the spawn-N-and-rejoin shape inherited from TypeScript.

Quality

Fuzz the text scanners (the overlay walker and top_level_text are classic targets). Add cargo-deny/cargo-audit — nothing currently watches the dependency tree's advisories. Adopt tracing behind a -v flag instead of bare println!. Once the parity check exists, turn the mini-flake blobs into golden fixtures so a schema change must consciously update both implementations in one commit.

The strategic opinion, briefly: keep the SPA in TypeScript forever, keep extract.nix as the single shared evaluator, and let the parity check — not enthusiasm — decide when the bun extractor retires.
